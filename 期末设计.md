    UpdatedAt     TIMESTAMP

    Id            SERIAL PRIMARY KEY,

    SupplierId    INT           REFERENCES Suppliers(Id),
    Unit          VARCHAR(20)   NOT NULL DEFAULT '个',

### 3.4 PurchaseService — 进货服务（核心）

#### 业务流程 1：新建进货单

```
用户操作：
  1. 选择供应商
  2. 动态添加商品行（JS 实现，每行选择商品 + 填数量）
  3. 提交

Service 层处理（CreateOrder）：
  1. 生成单号：格式 "JH" + yyyyMMdd + 三位流水号，如 JH20250101001
     ── 查询当天最大单号，流水号+1；若当天无单则从001开始
  2. 遍历明细行：
     a. 根据 ProductId 查询商品，取 PurchasePrice 作为 UnitPrice
     b. SubTotal = UnitPrice × Quantity
     c. 累加到 TotalAmount
  3. 主表 Status = "待入库"
  4. 主表 + 明细表一并写入数据库（此时库存不变）
```

#### 业务流程 2：确认入库

```
用户操作：在进货单列表点击"确认入库"

Service 层处理（ConfirmStockIn）：
  1. 校验：单据状态必须为"待入库"
  2. 开启数据库事务
  3. 遍历该单据的所有明细：
     a. 找到对应 Product
     b. StockQuantity += 本次进货 Quantity           ← 库存增加
     c. PurchasePrice = 本次进货 UnitPrice            ← 更新最新进货价
     d. UpdatedAt = 当前时间
  4. 主表 Status = "已入库"
  5. 提交事务
  6. 若任何一步失败，回滚事务并抛出异常
```

#### 业务流程 3：取消进货单

```
Service 层处理（CancelOrder）：
  1. 校验：状态必须为"待入库"（已入库的单据不可取消）
  2. Status = "已取消"
  3. 不涉及库存变动
```

### 3.5 SalesService — 销售服务（核心）

#### 业务流程 1：新建销售单

```
用户操作：
  1. 选择客户（可选，"散客"可不选）
  2. 动态添加商品行（JS，每行选择商品 + 填数量）
  3. 填入优惠金额（可选）
  4. 选择支付方式
  5. 提交

Service 层处理（CreateSale）：
  1. 生成单号：格式 "XS" + yyyyMMdd + 三位流水号
  2. 开启数据库事务
  3. 遍历明细行：
     a. 查询 Product，校验 StockQuantity ≥ 本次 Quantity  ← 库存不足则抛异常
     b. StockQuantity -= 本次 Quantity                    ← 库存扣减
     c. UpdatedAt = 当前时间
     d. UnitPrice = SellingPrice
     e. SubTotal = UnitPrice × Quantity
     f. 累加到 TotalAmount
  4. ActualAmount = TotalAmount - DiscountAmount
  5. 若选择了客户：
     a. Customer.TotalSpent += ActualAmount               ← 更新客户累计消费
     b. 根据 TotalSpent 自动调整 MemberLevel：
        - ≥ 10000 → "金卡"
        - ≥ 5000  → "银卡"
        - 其他    → "普通"
  6. Status = "已完成"
  7. 主表 + 明细表写入数据库
  8. 提交事务
  9. 若库存不足或其他步骤失败，回滚并抛异常
```

## 五、鉴权设计

### 5.1 方案

```
Session 鉴权
├── 登录成功 → Session["UserId"] = user.Id
│             Session["Role"]   = user.Role
│             Session["DisplayName"] = user.DisplayName
├── 每个请求 → 在 BaseController 或 Filter 中检查 Session["UserId"] 是否存在
│             不存在则跳转 /Account/Login
└── 管理员操作 → 额外检查 Session["Role"] == "管理员"
```

### 5.2 密码存储

```
存储方式：SHA256 哈希
用户输入密码 → SHA256(input) → 与数据库 PasswordHash 比对
```

## 八、单号生成规则

```
进货单：JH + yyyyMMdd + 三位流水号
  示例：JH20250101001、JH20250101002
  
销售单：XS + yyyyMMdd + 三位流水号
  示例：XS20250101001、XS20250101002

实现逻辑（在 Service 层）：
  1. 查询当天该类型最大单号
  2. 若存在，截取后三位转数字 +1，左侧补零
  3. 若不存在，从 001 开始
```

## 九、会员等级自动计算

```
每次销售完成后，根据 Customer.TotalSpent 重新判定等级：

  TotalSpent ≥ 10000  →  金卡
  TotalSpent ≥ 5000   →  银卡
  其他                →  普通

未来可扩展：不同等级享受不同折扣（答辩加分点）
```
